Запусти синхронизацию Beads в Anytype.

Используй агента `anytype-beads-sync` (subagent_type: `anytype-beads-sync`).

Если пользователь передал конкретные ID задач (например `$ARGUMENTS`), передай их агенту для **принудительной** инкрементальной синхронизации только этих задач (force-update без сравнения watermark).

Если аргументы пустые — выполни **инкрементальную полную синхронизацию** всех открытых задач Beads в Anytype space FundingBot, сравнивая watermark из mapping-файла. Обновляться должны ТОЛЬКО изменённые задачи.

Промпт для агента должен содержать:
1. Какие задачи синхронизировать (конкретные ID или все открытые) и режим (`force` для явных ID, `incremental` для пустых аргументов)
2. Напоминание: читать Beads через `bd list --json` / `bd show <id> --json`, писать в Anytype через MCP
3. Mapping-файл: `G:\Project-X\FundingBot\.claude\anytype-beads-mapping.json`

## КРИТИЧНО: ТОЛЬКО ОТКРЫТЫЕ ЗАДАЧИ

В Anytype синхронизируются **ТОЛЬКО** задачи со статусом `open` / не `closed`.
Задачи со статусом `closed` (и заполненным `closed_at`) **НЕ СОЗДАЮТСЯ** в Anytype и **УДАЛЯЮТСЯ** если были синхронизированы ранее.

## КРИТИЧНО: ИНКРЕМЕНТАЛЬНОСТЬ ПО WATERMARK (ПО УМОЛЧАНИЮ)

При полной синхронизации (`/anytype` без аргументов) агент работает ИНКРЕМЕНТАЛЬНО:
для каждой задачи Beads сравнивается тройка watermark `(updated_at, closed_at, status)`
с сохранённой в `anytype-beads-mapping.json`. **Обновляются ТОЛЬКО задачи с изменением хотя бы одного поля.** Совпадение всех трёх полей → SKIP, никакого MCP-вызова `update_object` не делается.

Это многократно ускоряет синк и экономит контекст. Без watermark-фильтра агент тратил `update_object` на каждую задачу маппинга, даже если в Beads ничего не менялось.

### Формат mapping-файла (v2, обязательный)

```json
{
  "FundingBot-07y": {
    "anytype_id": "bafyrei...xyz",
    "updated_at": "2026-04-27T12:34:56Z",
    "closed_at": null,
    "status": "open"
  }
}
```

Поля:
- `anytype_id` — ID объекта в Anytype (обязательно)
- `updated_at` — `updated_at` Beads-задачи на момент последнего успешного upsert (ISO 8601, UTC)
- `closed_at` — `closed_at` Beads-задачи или `null` если open
- `status` — `status` Beads-задачи на момент последнего upsert

### Обратная совместимость со старым форматом

Если значение для ключа — **строка** (старый формат `"beads_id": "anytype_id"`), агент трактует запись как STALE:
1. Безусловно обновляет объект в Anytype (UPDATE).
2. Пересохраняет запись в новом формате с актуальным watermark из Beads.
3. Дальнейшие синки этой задачи будут быстрыми (через сравнение watermark).

Это даёт одноразовый «прогон миграции» без дополнительного скрипта: первый `/anytype` после изменения схемы обновит все строковые записи; следующие — уже инкрементально.

## ПОРЯДОК СИНХРОНИЗАЦИИ (ОБЯЗАТЕЛЬНО)

При каждой полной синхронизации (`incremental` режим) агент ОБЯЗАН следовать этому порядку:

### Фаза 1: Сверка существующих маппингов
1. Прочитать `anytype-beads-mapping.json` — все маппинги `beads_id → entry` (entry может быть строкой v1 или объектом v2)
2. Прочитать актуальные задачи Beads (`bd list --all --json`) — построить map `beads_id → task`
3. Для каждого маппинга:
   - **Задача `closed`** или **closed_at заполнен** → УДАЛИТЬ объект из Anytype (`anytype_delete_object`), УДАЛИТЬ запись из маппинга
   - **Задача удалена из Beads** (нет в списке) → УДАЛИТЬ объект из Anytype, УДАЛИТЬ запись из маппинга
   - **Задача `open`, entry — строка (старый формат)** → пометить для **UPDATE** (миграция формата)
   - **Задача `open`, entry — объект v2, watermark совпадает (`updated_at == task.updated_at AND closed_at == task.closed_at AND status == task.status`)** → **SKIP**
   - **Задача `open`, entry — объект v2, watermark НЕ совпадает** → пометить для **UPDATE**
4. Записать обновлённый `anytype-beads-mapping.json` (после удалений)

### Фаза 2: Применение CREATE и UPDATE
1. Найти задачи со статусом `open`, которых НЕТ в маппинге → **CREATE**
2. Применить помеченные **UPDATE** из Фазы 1
3. После каждого успешного CREATE/UPDATE — записать в mapping entry в формате v2 с актуальным watermark из Beads-задачи
4. Атомарно перезаписать `anytype-beads-mapping.json`

### Фаза 1.5: Pre-sync summary (БЛОКИРУЮЩИЙ ШАГ)

Перед любым create/update/delete агент ОБЯЗАН вывести summary:

```
[anytype-sync] Pre-sync summary:
- Всего задач Beads (open): T
- К созданию: N (новые, без маппинга)
- К обновлению: M (watermark изменился или старый формат)
- Без изменений (SKIP): S (watermark совпадает) ← ключевая метрика инкрементальности
- Стейл-маппингов: K (нет в Beads — будут удалены)
- К удалению по closed_at: L (закрытые — будут удалены)
```

Если `M + N + K + L == 0` (только SKIP) — агент завершает работу с сообщением «Изменений нет, синк не требуется», без MCP-вызовов.

## ПРИНУДИТЕЛЬНЫЙ РЕЖИМ (force)

При вызове `/anytype <ID1> <ID2> ...` (с аргументами):
1. Получить статус каждой задачи через `bd show <id> --json`
2. Если задача `closed` → удалить из Anytype (если есть маппинг), удалить маппинг
3. Если задача `open` → **обновить безусловно** (без сравнения watermark — пользователь явно указал ID, значит хочет принудительный синк)
4. После успешного upsert — записать актуальный watermark в mapping

## КРИТИЧНО: ЗАЩИТА ОТ ДУБЛЕЙ

### Правило обновления объектов (ОБЯЗАТЕЛЬНО)
Anytype API при `update_object` с параметром `body`/`markdown` **ПЕРЕСОЗДАЁТ** объект с **НОВЫМ ID**.
Старый объект НЕ удаляется автоматически — это главная причина дублей.

**При обновлении существующего объекта (маппинг есть):**
1. Обновляй ТОЛЬКО `name` и `properties` (in-place, ID не меняется)
2. Если нужно обновить body/markdown:
   a. Сначала УДАЛИ старый объект через `anytype_delete_object(old_anytype_id)`
   b. Затем СОЗДАЙ новый через `anytype_create_object`
   c. Обнови маппинг с новым ID + watermark
3. НИКОГДА не вызывай `anytype_update_object` с параметром `body` или `markdown`

### Правило создания объектов (ОБЯЗАТЕЛЬНО)
Перед созданием нового объекта:
1. Проверь маппинг-файл — если `beads_id` уже есть, это UPDATE, не CREATE
2. Дополнительно проверь через `anytype_search_objects(query=beads_id)` — объект может существовать без маппинга
3. Если найден объект без маппинга — обнови маппинг (формат v2 с watermark) и сделай UPDATE вместо CREATE

## КРИТИЧНО: ПРАВИЛА МАППИНГА СВОЙСТВ ДАТА (ОБЯЗАТЕЛЬНО)

При синхронизации свойств-дат (`beads_opened_at` и т.д.):
- Если поле отсутствует в JSON или равно `null`/`undefined` — передавай `"date": null` (очистка поля), **НЕ подставляй текущую дату**

## ПАКЕТНАЯ ОБРАБОТКА

При большом количестве задач агент ОБЯЗАН обрабатывать их пакетами по 15 штук с промежуточным выводом прогресса. Если контекст заканчивается — остановиться и вывести список оставшихся ID для продолжения через `/anytype <ID1> <ID2> ...`.

В инкрементальном режиме SKIP-задачи **не считаются** в этих пакетах (по ним нет MCP-вызовов) — пакеты формируются только из CREATE/UPDATE/DELETE.
